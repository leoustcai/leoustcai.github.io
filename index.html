Execution flow

1. Intake stays familiar

The caller speaks an OpenAI-style shape, so adoption does not require a bespoke client protocol.

2. Dispatch becomes visible

The proxy writes an inbox task, which exposes routing rather than hiding it inside memory.

3. Runner owns the loop

The execution lifecycle stays with the runner, keeping the proxy thin and easier to reason about.

4. Specialists can hand off

One agent can route to another without losing workspace continuity, files, or current task state.

5. Shared context remains real

Agents act against the same filesystem, notes, and codebase instead of isolated sandboxes with no durable memory.

6. Response exits cleanly

The finished result returns through the operational chain and reaches the caller in a predictable format.

Why this model exists

Familiar public contract

Clients keep using a protocol they already understand while the internals remain agent-native.

Visible ownership

Inbox messages make routing, assignment, and responsibility legible once multiple agents collaborate.

Thin proxy boundary

The proxy handles intake, while execution complexity stays where it belongs: inside the runner and workspace.

Specialists over monoliths

Frontend, backend, and architecture work can split cleanly instead of collapsing into one opaque worker.

Durable shared context

The same workspace keeps files, notes, and active implementation aligned across handoffs.
