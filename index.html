Agent operations deck

This is intentionally not part of the homepage track above. It is a dedicated tabbed surface for the mechanics behind ai-agent-proxy and agent-runner, aimed at readers who want the request lifecycle in detail.

Why the split is strict

Most visitors need the product story first. The mechanics crowd needs the routing model, handoff chain, and return path. Combining both in one visual rhythm makes the page worse for everyone.

Familiar request intake Runner-owned handoff Predictable response path

What this deck covers

The tabs below split the material into a live map, a lifecycle explanation, and the architectural reasons this model is useful. The result feels like a technical deck with its own mood, not a leftover homepage section.

External caller

User / Client

Sends a chat-completions style request that already fits existing OpenAI or vLLM client habits.

API front door

ai-agent-proxy

Accepts the request, chooses the route, and writes a concrete task payload into the correct inbox.

Filesystem queue

Agent inbox

Messages land in a durable format that the selected runner can inspect, read, and act on.

Worker loop

Agent runner

Watches inbox folders, launches the assigned agent, and owns the task lifecycle until completion.

Specialist

Architect

Can reshape the task, split work, or set direction before specialist execution starts.

Specialist

Frontend engineer

Takes UI, interaction, layout, and presentation work when the request needs product-facing execution.

Specialist

Backend engineer

Takes server logic, automation, or systems work when the job belongs deeper in implementation.

Return path

Reply

The completed result is collected and sent back through the proxy as a normal API-shaped response.

Flow walkthrough

  1. 01
    Request hits the proxy The public contract stays familiar, so existing clients do not need a bespoke agent protocol.
  2. 02
    Proxy writes the task The request becomes a concrete inbox message targeted at the correct role or runner.
  3. 03
    Runner picks it up The long-lived work loop notices the message and starts the correct execution path.
  4. 04
    Specialists can hand off The active agent can route work across architect, frontend, backend, or other specialists in the same workspace.
  5. 05
    Result returns cleanly The response exits through the same operational chain, so the caller sees a normal API result.

Legend

Request Handoff Response
1. Intake

The caller sends an OpenAI-style request to the proxy, which keeps the public API familiar.

2. Dispatch

The proxy writes a task into an inbox, which makes routing visible instead of hiding it in transient state.

3. Execution

The runner owns the long-lived work loop and can start the right specialist path without mixing concerns into the proxy.

4. Shared context

Agents operate in the same filesystem context, which keeps files, notes, and implementation state aligned.

5. Handoff

One agent can delegate to another without losing continuity, which matters once tasks become genuinely multi-step.

6. Return path

The completed result comes back through a predictable chain rather than being manually stitched together afterward.
