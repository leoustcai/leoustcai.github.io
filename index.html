AI Agent Proxy + Agent Runner working diagram

An interactive view of the flow you described: a user sends an OpenAI-style or vLLM-style API request to ai-agent-proxy, the proxy writes a task into an agent inbox, the agent runner picks it up, the team can hand work across agents, and the reply returns through the same chain.

External caller

User / Client

Sends a chat-completions style request to the proxy, compatible with OpenAI or vLLM-style patterns.

API front door

ai-agent-proxy

Accepts the API request, maps it to an agent, and writes a task message into that agent’s inbox.

Task queue

Agent inbox

A filesystem inbox where work lands as messages for the selected agent or role.

Worker loop

Agent runner

Watches inbox folders, picks up tasks, launches the assigned agent, and manages the work lifecycle.

Team agent

Architect

Can plan, split, redirect, or reshape the task before passing it deeper into the team.

Team agent

Frontend engineer

Handles UI, diagrams, pages, interactive surfaces, and presentation logic when that work is handed off.

Team agent

Backend engineer

Handles server behavior, logic, and implementation details when the task needs backend execution.

Response path

Reply

The completed result is written back, collected by the proxy layer, and returned to the original caller.

Mechanism

This page is powered by JavaScript. The diagram highlights the path step-by-step so the flow is readable instead of static.

Request flow Team handoff Response flow
Flow walkthrough
  1. 01
    Request enters the proxy User sends an OpenAI-style or vLLM-like API request to ai-agent-proxy.
  2. 02
    Proxy writes to inbox The proxy converts the request into a task message and places it in the chosen agent inbox.
  3. 03
    Runner picks it up The agent runner watches that inbox, reads the task, and launches the assigned agent flow.
  4. 04
    Agent team handoff The active agent can hand work to other team agents however it wants: architect, frontend, backend, or others.
  5. 05
    Result comes back The completed output returns through the runner/proxy path and goes back to the caller as the API response.
Why this matters
  • The API surface stays familiar to clients that already speak OpenAI-style chat APIs.
  • The proxy and runner separate request intake from actual agent execution.
  • The inbox layer makes dispatch visible, inspectable, and easy to reason about.
  • Team handoff lets one agent orchestrate specialists instead of forcing a single monolithic worker.
