What this section explains

The lab is split into three tabs: a working diagram, the lifecycle in plain language, and the architectural reasons this model is useful. It keeps the explanation specific instead of dumping all AI content into the homepage flow.

External caller

User / Client

Sends a chat-completions style request that looks familiar to OpenAI or vLLM clients.

API front door

ai-agent-proxy

Accepts the request, selects the target agent path, and writes a task payload into the correct inbox.

Task queue

Agent inbox

A filesystem inbox where messages land in a format the selected runner can inspect and act on.

Worker loop

Agent runner

Watches inbox folders, launches the assigned agent, tracks lifecycle, and keeps the work moving.

Team agent

Architect

Can reshape or decompose the task before passing execution deeper into the team.

Team agent

Frontend engineer

Takes UI, page structure, presentation logic, and interaction work when the task requires it.

Team agent

Backend engineer

Takes implementation work that belongs in server logic, automation, or systems code.

Response path

Reply

The completed result is collected and returned back through the proxy to the original caller.

Flow walkthrough

  1. 01
    Request hits the proxy The caller sends a standard API-shaped request, so existing clients do not need a strange new protocol.
  2. 02
    Proxy writes the task The proxy converts the request into a concrete inbox message for the selected role or agent.
  3. 03
    Runner picks it up The runner loop notices the new item, reads it, and starts the matching agent workflow.
  4. 04
    Specialists can hand off The active agent can route work across architect, frontend, backend, or other specialist runners.
  5. 05
    Result returns cleanly The response comes back through the same operational chain, so the caller sees a normal API response.

Legend

Request Handoff Response
1. Intake

The public contract stays familiar: users send an OpenAI-style request to the proxy instead of learning a bespoke control protocol.

2. Dispatch

The proxy writes a task into an inbox, which makes the routing visible and inspectable instead of disappearing into hidden process state.

3. Execution

The runner owns the long-lived work loop and can choose the correct specialist or team structure for the job.

4. Handoff

One agent can delegate to others without losing the shared workspace context that makes the task coherent.

5. Return path

The completed result is written back through a predictable path rather than being stitched together ad hoc.

6. Reader takeaway

The system is easier to reason about because intake, execution, and response are separated into concrete layers.

Familiar API surface

Clients can keep speaking the protocol they already know while the internal implementation stays agent-native.

Visible handoff

Inbox messages make task routing and ownership inspectable, which matters when several agents are collaborating.

Runner as control plane

The runner centralizes the work loop, so the proxy does not become overloaded with execution concerns.

Specialists over monoliths

The model allows one task to move across architect, frontend, backend, or other roles without flattening everything into one worker.

Shared workspace continuity

Agents operate in the same filesystem context, which keeps files, notes, and state aligned with the actual project.

Homepage discipline

Putting this explanation here, instead of in the hero, keeps the main site readable for visitors who are not here for protocol mechanics.
